# DECISIONS.md - personal-trainer

## Decisoes tecnicas

| Decisao | Motivo |
|---|---|
| Functions Firebase com service accounts dedicadas | `api` e `claudeWorker` nao devem depender da Compute default service account nem de `roles/editor`. O deploy define defaults duraveis (`pt-api-runtime` e `pt-worker`) com override por env; `api` permanece publica por IAM porque e a entrada HTTP protegida por auth app-level, enquanto `claudeWorker` fica invocavel apenas pela service account OIDC das Cloud Tasks e ainda valida OIDC no codigo. |
| `BOOTSTRAP_SECRET` via Secret Manager e CSP em enforcement | Segredo de bootstrap deixa de depender de env plain em Cloud Functions e passa a ser vinculado como secret apenas na Function `api`. O Hosting e o backend passam a usar CSP em enforcement por padrao; o registro do service worker saiu do script inline para `public/sw-register.js`, permitindo remover `script-src 'unsafe-inline'` sem quebrar a estrategia de atualizacao online. |
| `replace_all` em `file=plano` permitido em qualquer conversa | Revoga parcialmente a decisão de 2026-02-27 (bloqueio em chat/edição). A IA agora pode reescrever o plano da data alvo tanto em conversa de plano (continuação) quanto em conversa geral (auto-log, "comi X", "atualize meu plano"). Proteção delegada às camadas existentes: (a) `planDateLock` sempre setado (= `planDate` do contexto da conversa OU data exibida no app) trava o escopo ao único dia; (b) `enforcePlanUserCheckedPermission` continua exigindo card de permissão para alterar itens marcados pela usuária; (c) heurística `buildCoachNoteUpdateFromReplaceAll` reduz blast radius automaticamente quando só a nota mudou. Motivado pelo caso 2026-05-15 em que a IA emitia `replace_all` legítimo para trocar um item ("não quero omelete") e o defensive default dropava silenciosamente. |
| Intent (`autoAction`/`conversationType`/`planDate`) persistido no `pendingResponses` | Em modo Firebase a resposta da IA volta via polling do `pendingResponses` (Cloud Tasks), num ciclo onde o `currentConvoMeta` do client pode ter mudado entre o POST e o retorno. Persistir os 3 campos como top-level no doc do pending preserva o contexto da request (data alvo, tipo de conversa, intent de auto-trigger) sem depender do estado volátil do app. `planDate` agora também é persistido em conversa geral, capturando a data exibida no app no momento do POST. |
| `normalizePlanDay` reutilizado por DocsContext e planUpdateGuard | Gemini Flash emite `nota_coach` (snake_case) no JSON do plano enquanto o resto do app usa `notaCoach` (camelCase). Centralizar a conversão num único util (`src/utils/planNormalize.js`) garante que tanto a persistência (DocsContext) quanto a comparação para detecção de note-only (planUpdateGuard.serializeComparablePlan) operem sobre a mesma forma canônica. Evita notas órfãs no Firestore e regressões da heurística `buildCoachNoteUpdateFromReplaceAll`. |
| Drops do guard com `console.warn` estruturado | Cada `return null` em `lockPlanUpdateToDate` passa por `dropUpdate(reason, update)` antes — registra `{reason, file, action, targetDate}` no console com prefix `[planUpdateGuard]`. Razões enumeradas: `replace_all_payload_invalid_or_date_mismatch`, `replace_all_not_authorized`, `granular_payload_invalid`. Custo de 1 linha de log é menor do que de uma regressão silenciosa como a do hotfix 2026-05-15. |
| Firebase backend paralelo por flag | `FIREBASE_BACKEND=true` permite desenvolver Hosting/Functions/Auth/Firestore sem quebrar o runtime SQLite da VPS |
| Firestore recria conceitos, sem importar SQLite automaticamente | Cutover sera tratado como re-onboarding, preservando escolha de nao migrar dados existentes por padrao |
| Response Inbox assincrona via Cloud Tasks | Evita timeout de Hosting/Functions em chamadas longas de IA e preserva recuperacao apos reload |
| Worker Firebase dedicado | Cloud Tasks chama somente a Function `claudeWorker` com timeout longo; a API HTTP curta nao monta rota interna de worker |
| OIDC entre worker e ai-gateway | Worker envia ID token ao gateway e o gateway deve validar service account/audience antes de queimar quota Vertex |
| Anexos multimodais via `mediaRef` privado | Upload de imagem/camera/audio vai para Cloud Storage via backend; conversas, pendingResponses e aiLogs guardam apenas metadados e `mediaRef`. O `gs://` e resolvido no worker somente no payload efemero ao `ai-gateway`, reduzindo vazamento de base64, custo de Firestore e tamanho de contexto. Upload tambem tem cota de objetos/bytes pendentes por usuario, delete explicito para composer, TTL em `mediaUploads.expiresAt` e audio restrito a WAV com duracao verificavel para limitar abuso sem envio ao chat. |
| Firestore state docs para invariantes | Conversa atual e contador de versoes de plano usam documentos determinísticos em transações, em vez de depender de queries concorrentes |
| Reclaim de `in_flight` antigo | PendingResponses em processamento podem ser reclamadas apos `max(CLOUD_TASKS_INFLIGHT_STALE_MS, 1.5 * GATEWAY_TIMEOUT_MS)`, evitando orfaos sem duplicar chamada Vertex ainda viva |
| Timeout do gateway como erro transitorio | Timeout/falha de rede no worker mantem `pendingResponses` em `in_flight` para retry da Cloud Task; apenas erro deterministico vira `failed` |
| Rate limit Firebase via Firestore | `/api/claude` em modo Firebase usa contador transacional compartilhado entre instancias, com TTL em `rateLimits.expiresAt`, nao memoria local da Function |
| Admin via Firebase Auth custom claims | Autorizacao sensivel fica no token validado pelo backend; espelho em perfil serve apenas UI/metadado |
| `AI_MODEL=gemini-3-flash` ou default do gateway | Garante que o personal-trainer use Gemini no gateway Firebase sem depender do default local Claude CLI |
| Inline styles + CSS classes | Compatibilidade com tema dinamico + CSS variables |
| `window.storage` abstraction | Permite fallback localStorage quando offline |
| Backend proxy via ai-gateway | Centralizar chamadas Claude, multi-provider e proteger API key |
| Extended thinking habilitado | Melhor qualidade de resposta para coaching |
| Structured outputs (`json_schema`) | Garante JSON valido sem parsing manual |
| `plano` como fonte canonica do dia | Checkboxes, nutricao e treino realizado passam a nascer do plano; `cal` e `treinos` viram projeções/cache |
| 9 documentos separados | Granularidade de edicao e persistencia |
| `is_current` flag em conversations | Separa conversa ativa de arquivadas |
| Seed defaults no setup | Runtime SQLite mantem contexto inicial da Renata; Firebase bootstrap usa documentos vazios por padrao e so usa seed Renata com `FIREBASE_BOOTSTRAP_SEED=renata` |
| `perfil.treinos_planejados` como agenda semanal | Evita duplicidade de fonte entre Perfil e Saúde |
| `DocsContext` com store central e mutacoes coordenadas | Evita falha silenciosa e centraliza rebuild de projeções derivadas |
| `/api/health` como health operacional unico | Mantem simplicidade da VPS e ainda verifica SQLite + gateway |
| `calorias` e `treinos` removidos do enum do schema | Impede que a IA envie updates diretos para documentos derivados; reforça que saúde nasce do plano |
| Frontend guard para updates derivados | `DocsContext` descarta silenciosamente updates com file=calorias/treinos como safety net |
| Auto-log obrigatório no prompt | Quando o usuario reporta comida/treino, a IA deve registrar no plano imediatamente (append_item/patch_item), sem esperar pedido explícito |
| Data em conversa geral = data do app | Em conversas general, "meu plano" refere-se a data exibida no app (plan_context.date), nao amanha |
| "Plano" = aba Plano, não conceito genérico | Quando usuario fala "atualize o plano", instrução explícita no prompt para gerar update file=plano (replace_all), não confundir com atualizar perfil/memoria |
| Safe JSON parse em `applySingleUpdateToDocs` | 7 `JSON.parse` substituidos por `parseJson(value, null)` com fallback gracioso — previne crash por JSON malformado da IA |
| Error handling em `rebuildHealthCache` | Adicionado try/catch + reload no rebuild de cache de saude — previne state divergence se persist falhar |
| Reply vazia rejeitada no parser | `claudeResponseParser` agora rejeita `reply: ""` como INVALID_SCHEMA |
| Health check usa `response.ok` | Aceita apenas 200-299 do gateway (antes aceitava 4xx/5xx) |
| Senha minima 6 chars | Aumentado de 4 para 6 caracteres minimos no setup/registro |
| CLI retry com backoff 500ms | Delay antes de retry e guard contra retry duplo no 410 CLI_SESSION_EXPIRED |
| Content-Type enforcement em `/api/*` | POST/PUT/DELETE com body devem enviar application/json — protecao CSRF basica |
| Audit logging em auth | Login OK/falho e logout registrados com JSON estruturado no console |
| Validacao `doc_key` e `messages.length` | doc_key regex `/^[a-z0-9_-]{1,50}$/`; messages limitado a 100 por request |
| Redaction expandida em AI logs | Set de chaves redactadas inclui token, api_key, apiKey, cookie |
| `<raw_data>` removido do contexto | calorias_json e treinos_json removidos do prompt — dados ja derivados do plano e presentes em `<nutrition_today>` |
| `planScopeDate` fallback para hoje | Em conversa plan sem planDate, schema usa data de hoje como fallback |
| `maxItems: 30` no schema updates | Limita array de updates a 30 itens por resposta da IA |
| Onboarding usa acoes incrementais | replace_all apenas na consolidacao final; turns intermediarios usam append/patch |
| Validacao nutricional no prompt | IA deve confirmar com usuario se item reportar >1500kcal ou >100g proteina |
| `normalizeGroupName` com accent stripping | Matching de grupo de refeicao insensivel a acentos; fallback para "Outros" |
| Session sliding window | Sessao estendida quando ultrapassar metade do TTL — usuario ativo nao perde sessao |
| Rate limit no ai-gateway | 30 req/min em `/api/chat` via express-rate-limit |
| Validacao de interaction_context | Tipo string enforced e truncado em 10000 chars no gateway |
| Documento `medidas` (10º doc_key) | Time-series estruturado para medições corporais (peso, gordura, TMB, circunferências) — substitui dados numéricos no historico |
| `add_medida` action no schema | Permite IA registrar medições via structured output; historico passa a ser apenas qualitativo |
| Diff automático no save do Perfil | Mudança em peso/gordura/TMB cria entrada em medidas; mudança de metas cria progresso (Mudança de fase) |
| `macros_alvo` exposto na UI | Metas nutricionais diárias visíveis e editáveis no Perfil — antes era dado oculto |
| `preferencias_alimentares` exposto na UI | Preferências alimentares editáveis via TagEditor no Perfil — antes invisível ao usuário |
| `notaCoach` visível no PlanoView | Nota diária do coach expansível (read-only) no rodapé do plano — antes explicitamente oculta |
| Migração bootstrap de medidas | Usuários existentes ganham medidas semeado do perfil no primeiro load via serializeDocsFromResponse |
| Gráfico SVG puro para peso | WeightTrendChart sem dependências externas — linha + faixa de meta + pontos |
| NovaMedicaoForm inline na Saúde | Formulário expansível para registrar medições diretamente na aba Saúde |
| Gatilhos explícitos de progresso no prompt | IA recebe lista concreta de quando criar cada tipo (Conquista, Obstáculo, Dificuldade, Mudança de fase) |
| TagEditor como componente reutilizável | Componente de tags editáveis para arrays de strings — usado em preferências alimentares |
| Range validation em `add_medida` | peso 0-300, gordura 0-80, tmb 0-10000 — previne dados absurdos da IA |
| Tolerância numérica no `diffPerfil` (0.01) | Evita falsos positivos de bodyChanged quando perfil é salvo com arredondamento diferente (string vs number) |
| Dedup de medidas por data+metodo em 3 pontos | Previne entradas duplicadas em savePerfil (perfil), addMedida (form), add_medida (IA) |
| Limite 365 entradas em medidas (FIFO) | Evita crescimento ilimitado; entradas mais antigas descartadas automaticamente |
| `isExternalUpdate` ref no PerfilTab | Previne auto-save cascade quando perfil é atualizado externamente (sync da IA ou formulário da Saúde) |
| Schema `content` aceita string ou object | Handler já suportava ambos; schema atualizado para refletir realidade (`type: ["string", "object"]`) |
| Sanitização de notas em medidas | HTML tags removidas e notas limitadas a 500 chars — defense-in-depth contra XSS |
| `macros_alvo` e `preferencias_alimentares` no contexto da IA | Dados editáveis pelo usuário agora visíveis para a IA no user_profile |
| Number() coercion no WeightTrendChart | Previne NaN em SVG quando peso_kg é string; guard metaMin<=metaMax na banda de meta |
| Erros de API enriquecidos com `statusCode`/`code`/`payload` | `services/api.js` anexa metadados ao Error em todos os 4 verbos; `error.message` continua igual (mudança aditiva). Permite frontend mapear backend errors para mensagens user-friendly específicas sem string matching frágil |
| Códigos estruturados em `auth/auto-register` (`WHITELIST_MISS`, `MISSING_EMAIL`, `INTERNAL`) | Backend distingue tipos de falha em login Google. Mensagem do usuário em `WHITELIST_MISS` revela explicitamente que o email não está autorizado — trade-off de info disclosure aceito porque atacante já consegue testar via UI Firebase Auth, e a mensagem genérica frustra usuários legítimos |
| Log JSON estruturado para `event:"auth.failure"` | `console.log(JSON.stringify({event,reason,uid,email,ts}))` separado do `console.error` legado permite filtrar no Cloud Logging sem perder stack traces. Tipos: `whitelist_miss`, `missing_email`, `internal` |
| Onboarding em 3 telas com save atômico no fim | Tela 1 (nome obrigatório) → Tela 2 (objetivo radio + descrição opcional) → Tela 3 (restrições checkboxes + frequência treino + tipo). `OnboardingFlow.jsx` mantém state, chama `onSave` apenas na conclusão da Tela 3. Botão "Pular" em telas 2 e 3 preserva campos parciais (incluindo `outras` typed mas não-elevado). Reduz drop-off vs flow obrigatório |
| Onboarding NÃO popula `treinos_planejados` direto | Frequência + tipo coletados na Tela 3 viram texto em `meta_descricao` (`"Treina atualmente 3-5x por semana (musculação)."`). `treinos_planejados` mantém shape `{dia,tipo,duracao,horario}` populado pela IA via chat com permissão — evita shape divergente quebrando o prompt |
| `isFirstSession` testa `!(idade && peso_kg)` | Heurística do welcome adaptativo do ChatTab passa a olhar campos que SÓ o chat captura (idade, peso). Mantém welcome "Prazer em te conhecer" pós-onboarding até IA completar descoberta corporal, transitando depois para welcome normal |
| Token budget hard cap diário + mensal por usuário | `TOKEN_BUDGET_DAILY=500000` (~$0,05/dia em gemini-3-flash) e `TOKEN_BUDGET_MONTHLY=5000000` (~$0,50/mês) previnem custo runaway por bug ou abuso. Override por env, fail-open em erro de leitura para não bloquear chat por bug de Firestore |
| Token budget tambem registra modalidade/custo estimado | O worker debita `textTokens`, `imageTokens`, `audioTokens`, contagem de imagens, segundos de audio e `estimatedCostMicros`. `TOKEN_BUDGET_DAILY_COST_MICROS`/`TOKEN_BUDGET_MONTHLY_COST_MICROS` sao caps opcionais, desativados por padrao, para segurar audio/imagem quando token bruto nao refletir bem custo real. Os multiplicadores `COST_TEXT_INPUT_MICROS_PER_TOKEN`, `COST_IMAGE_INPUT_MICROS_PER_TOKEN`, `COST_AUDIO_INPUT_MICROS_PER_TOKEN` e `COST_OUTPUT_MICROS_PER_TOKEN` deixam a estimativa ajustavel por modelo/preco sem alterar codigo. O pre-check soma a estimativa multimodal da request atual antes de enfileirar Cloud Task. |
| Token budget em subcollection `users/{uid}/tokenBudgets/{period}` | Subcollection (não raiz) mantém consistência com `aiLogs`/`conversations`/`pendingResponses`. TTL via `expiresAt` (48h daily / 35d monthly) limpa docs velhos sem cron manual |
| Débito atômico via `FieldValue.increment` | Worker debita após `response.ok` em paralelo (Promise.all daily+monthly). `set({...}, {merge:true})` cria doc na primeira vez sem race condition. Aceita overshoot de 1 request quando task já está in-flight no momento do bloqueio |
| 429 com `code:"TOKEN_BUDGET_EXCEEDED"` em pre-check | Pre-check em `routes/firebaseClaude.js` ANTES de enfileirar Cloud Task evita queimar quota do Vertex em request que será rejeitada. `error.message` já user-friendly em pt-BR ("Você atingiu o limite diário..."), sem necessidade de mapping no frontend |
| CI/CD via GitHub Actions com `environment: production` | Deploy a `main` exige aprovação manual via required reviewer no environment, evitando push acidental sem revisão. Lint+test+build rodam em qualquer branch (paralelo). Smoke pós-deploy bate `/api/health`; falha vermelha alerta o autor |
| `--only "hosting,firestore,storage,functions:api,functions:claudeWorker"` no CI | Deploy seletivo publica Hosting, Firestore rules/indexes, Storage rules e Functions do PT sem tocar a função `gateway` que vive no sub-projeto `ai-gateway` (Firebase project compartilhado). Repete o mesmo padrão do deploy manual e protege contra a regressão histórica de 2026-05-08 |
| `VITE_BASE_PATH` lido via `loadEnv` em `vite.config.js` + `.env.production` versionado | Default `/pt/` mantém dev local via Caddy proxy. `.env.production` versionado com `VITE_BASE_PATH=/` garante que `vite build` para Firebase Hosting (que serve em raiz) sempre saia com paths absolutos corretos, sem depender de quem lembra de exportar a var no shell. Previne regressão da tela-em-branco observada após Sprint 2/3 |
| `index.html` usa `%BASE_URL%` em manifest/sw/icon paths | Tags `<link rel="manifest">`, `<link rel="icon">` e o `navigator.serviceWorker.register` ficam agnósticos ao base path. Vite substitui `%BASE_URL%` no build (dev=`/pt/`, prod=`/`) — mesmo HTML serve dev e prod sem hardcode |
| Sharp em devDep para gerar ícones PWA | Sharp (libvips pré-compilada) é cross-platform (Windows dev + Linux CI), tem controle programático de canvas/safe-zone e binário automático em `npm ci`. ImageMagick exige PATH externo, Inkscape exige instalação manual. Script `npm run icons:generate` é manual — não roda em prebuild para não atrasar `npm run build` em todo CI run; só roda quando `icon.svg` muda |
| Maskable icon em entry separada (não combinado `purpose: "any maskable"`) | Combinar `any maskable` num único entry tem comportamento ambíguo entre Chrome/Edge/Firefox (alguns aplicam máscara onde não deveriam, outros não aplicam onde deveriam). 4 entries explícitas (SVG `any`, PNG-192 `any`, PNG-512 `any`, PNG-512 `maskable`) eliminam ambiguidade. Maskable usa safe-zone 80% com bg `#B87850` (= theme_color) para continuidade visual quando Android aplica adaptive mask |
| `manifest.id: "pt-coach"` como string fixa, não derivado de base | `id` é identidade estável que o browser usa para reconhecer "mesma PWA" entre versões — não pode mudar entre dev (`/pt/`) e prod (`/`). Fixar em string distintiva (`pt-coach`) evita perda de install state em rotação de base path |
| SW broadcast `SW_UPDATED` + `SKIP_WAITING` para refresh ativo | `public/sw.js` v5 dispara `SW_UPDATED` em `activate` (apenas se havia caches antigos — gate `oldCaches.length > 0` evita falso positivo em primeira instalação). Cliente expõe `UpdateBanner` "Nova versão disponível". Clique chama `registration.waiting?.postMessage({type:"SKIP_WAITING"})` + `location.reload()`. Padrão reaproveitado do DS-Dashboard `santuario-v11` |
| `register({ updateViaCache: "none" })` para SW | Sem essa flag, `sw.js` cacheia 24h via HTTP, atrasando detecção de versão nova. Com `none`, browser revalida em toda visita; combinado com `Cache-Control: no-cache` em firebase.json, propagação fica em segundos |
| Plugin Vite inline `pwaBaseTransform` em `vite.config.js` | Vite só substitui `%BASE_URL%` automaticamente em `index.html`; arquivos em `public/` são copiados as-is. Plugin com hook `closeBundle` substitui `__BASE__` por `base` resolvido em `dist/manifest.json` e `dist/sw.js` após build. Inline (vs script standalone `scripts/build-pwa.mjs`) mantém transformação atômica com `vite build`. Em dev `closeBundle` não roda — manifest é servido com `__BASE__` literal, aceitável (PWA validation só faz sentido contra prod build) |
| Cache headers PWA em `firebase.json` | `/manifest.json` e `/sw.js` com `no-cache` propagam mudanças em segundos (sem TTL CDN longo). `/icons/**` com `immutable + 1 ano` para performance — versionamento controlado pelo manifest, não pelo filename hash |
| `useInstallPrompt` segue padrão de `usePendingRecovery` | State local + listeners no `useEffect` com cleanup determinístico. `beforeinstallprompt` event guardado em `useRef` (evita re-renders durante captura). `dismiss` em `localStorage` com TTL 7 dias evita cansar usuária com prompts repetidos |
| `InstallButton` em `src/components/ui/` (não em onboarding ou layout) | `ui/` é a subdir genérica/agnóstica do design system. Outras subdirs (`chat/`, `onboarding/`, `perfil/`, `saude/`, `layout/`) são domain-specific. Componente é renderizado em SetupForm/LoginForm/RegisterForm — telas de entrada, antes do app fitness denso |
| iOS instalação via instrução visual (Share → Add to Home Screen) | Safari iOS não dispara `beforeinstallprompt` (até iOS 17.4). Único caminho é Share Sheet manual. `InstallButton` detecta `isIos` via UA e renderiza instrução visual com ícone Share inline (sem dependência externa). Trade-off aceito: UX inferior em iOS, mas viável sem hack de in-app browser |
| `registration.update()` em 4 gatilhos client-side (focus, visibilitychange, online, setInterval 5min) | Browser nao revalida sw.js sozinho durante sessao longa. Uma aba aberta por horas pode segurar versao antiga mesmo com `updateViaCache:"none"`. Os 4 gatilhos cobrem: alt-tab/switch de app (focus), aba volta a foreground (visibilitychange), reconectou pos-offline (online), idle prolongado (timer). Cada chamada eh guard-rail por `navigator.onLine` para nao desperdicar bandwidth em modo aviao |
| Listener `updatefound` + `statechange === "installed"` para banner imediato | Sem isso, UpdateBanner so aparece quando o novo SW completa `activate` (que so acontece quando todas as abas controladas fecham, ou via `skipWaiting`). Com o listener, o banner aparece assim que o novo SW termina `install` e entra em `waiting` — usuaria ve "Nova versao disponivel" em segundos apos o deploy, nao minutos/horas |
| `firebase.json`: `/index.html` e `/` com `no-cache`; `/assets/**` com `immutable, 1ano` | `index.html` cacheado no CDN seguraria as referencias aos hashes velhos do bundle, neutralizando o `vite build` (o usuario receberia bundle antigo mesmo com CDN apontando para arquivo novo). Bundle Vite tem hash no filename (`index-OWsuTCu6.js`) — quando muda eh um novo arquivo, entao cache imutavel eh seguro e elimina round-trips de revalidacao |
